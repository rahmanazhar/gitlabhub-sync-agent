# GitHubLab Sync

> Bidirectional mirror and pull-request sync between **GitHub** and **GitLab** — keep two providers as live double backups of each other.

[![CI](https://github.com/rahmanazhar/gitlabhub-sync-agent/actions/workflows/ci.yml/badge.svg)](https://github.com/rahmanazhar/gitlabhub-sync-agent/actions/workflows/ci.yml)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Python 3.9+](https://img.shields.io/badge/python-3.9%2B-blue.svg)](https://www.python.org/)

GitHubLab Sync keeps your repositories alive on **both** GitHub and GitLab. If a
repository only exists on one side, it is **created on the other** automatically.
Branches and tags are synced in both directions, and open **pull requests are
mirrored to merge requests** (and vice-versa). It is built for redundancy: lose
access to one provider and you still have a complete, up-to-date copy on the other.

It ships in two forms:

- **A standalone CLI** (`githublab-sync`) — packaged, testable, and runnable
  unattended from cron or CI for true hands-off double backup.
- **A Claude Code subagent** (`.claude/agents/repo-sync.md`) — drives the CLI
  interactively, previews changes, and helps resolve diverged branches.

## Features

- 🔁 **Bidirectional sync** — branches and tags flow both ways, fast-forward only.
- 🪞 **Auto-mirror creation** — missing repos are created on the other provider.
- 🔀 **PR ⇄ MR sync** — open pull requests and merge requests are mirrored across.
- 🛡️ **Safe by default** — diverged branches are *reported*, never force-overwritten.
- 🧪 **`--dry-run`** — preview every action before anything changes.
- 🧹 **No AI attribution** — strips `Co-Authored-By: Claude` / "generated by" markers from mirrored PR/MR text; nothing it writes references AI tooling.
- 🩺 **`doctor`** — one command to validate tokens, namespaces and connectivity.
- 🏢 **Self-hosted friendly** — works with GitHub Enterprise and self-managed GitLab via `host` / `api_url`.

## How it works

For each configured repository, GitHubLab Sync maintains a **bare cache repo**
and fetches both providers into separate ref namespaces so nothing collides:

```
                 ┌─────────────┐         ┌─────────────┐
   GitHub  ─────▶│             │◀─────   GitLab
   (fetch)       │  bare cache │  (fetch)
                 │  repository │
   GitHub  ◀─────│             │─────▶   GitLab
   (push FF)     └─────────────┘  (push FF)
```

Per branch, it compares the two sides:

| Situation | Action |
|---|---|
| Exists on one side only | Create it on the other |
| One side strictly ahead (fast-forward) | Push it to the other |
| Identical | Nothing to do |
| **Diverged** (both have unique commits) | **Report a conflict, change nothing** |

Tags are propagated when missing on a side. Pull requests / merge requests are
matched by `(source_branch, target_branch)` and recreated where absent.

## Quick start

```bash
# 1. Provide credentials (see "Tokens & permissions" below)
export GITHUB_TOKEN=ghp_...
export GITLAB_TOKEN=glpat-...

# 2. Generate a config and edit it
githublab-sync init
$EDITOR githublab-sync.yaml

# 3. Verify everything is reachable
githublab-sync doctor

# 4. Preview, then sync for real
githublab-sync sync --dry-run
githublab-sync sync
```

## Configuration

`githublab-sync init` writes a starter `githublab-sync.yaml`. Full example
([githublab-sync.example.yaml](githublab-sync.example.yaml)):

```yaml
github:
  owner: your-github-user-or-org
  token: ${GITHUB_TOKEN}

gitlab:
  owner: your-gitlab-group-or-user
  token: ${GITLAB_TOKEN}

sync:
  direction: bidirectional      # bidirectional | github-to-gitlab | gitlab-to-github
  create_missing: true
  sync_branches: true
  sync_tags: true
  sync_pull_requests: true
  strip_ai_attribution: true
  default_visibility: private

repositories:
  - my-service                  # same name on both providers
  - name: website               # different names per provider
    github_name: company-website
    gitlab_name: website-mirror
```

`${VAR}` placeholders are expanded from the environment, so **no secrets live in
the file** — it is safe to commit (the real `githublab-sync.yaml` is git-ignored
by default; commit `githublab-sync.example.yaml` instead).

### Options reference

| Key | Default | Description |
|---|---|---|
| `<provider>.owner` | — | GitHub user/org or GitLab namespace path (**required**) |
| `<provider>.token` | — | API token, typically `${GITHUB_TOKEN}` / `${GITLAB_TOKEN}` |
| `<provider>.host` | `github.com` / `gitlab.com` | Host for Enterprise / self-managed |
| `<provider>.api_url` | provider default | REST API base URL |
| `sync.direction` | `bidirectional` | Sync direction |
| `sync.create_missing` | `true` | Create the repo on a provider if absent |
| `sync.sync_branches` | `true` | Sync branches |
| `sync.sync_tags` | `true` | Sync tags |
| `sync.sync_pull_requests` | `true` | Mirror open PRs ⇄ MRs |
| `sync.strip_ai_attribution` | `true` | Remove AI authorship markers from PR/MR text |
| `sync.default_visibility` | `private` | Visibility for newly created mirrors |
| `sync.cache_dir` | `~/.cache/githublab-sync` | Where bare cache repos live |

## Tokens & permissions

| Provider | Token type | Scopes |
|---|---|---|
| **GitHub** | Personal Access Token (classic or fine-grained), or App installation token | `repo` (contents + pull requests). Org repos also need `admin:org` to create. |
| **GitLab** | Personal / group / project access token | `api`, `write_repository` |

Provide them via environment variables (`GITHUB_TOKEN`, `GITLAB_TOKEN`) or a
`.env` file (see [.env.example](.env.example)). **Never commit real tokens.**

## Usage

```
githublab-sync init [--config PATH] [--force]
githublab-sync doctor [--config PATH] [-v]
githublab-sync sync   [--config PATH] [--dry-run] [--repo NAME ...] [--direction DIR] [-v]
```

- `--dry-run` — print every intended action, change nothing.
- `--repo NAME` — limit to specific repositories (repeatable).
- `--direction` — override the configured direction for this run.

**Exit codes:** `0` success · `1` errors occurred · `2` config/usage error ·
`3` clean run but branch conflicts need human review.

## Running unattended

A scheduled GitHub Actions workflow is included at
[.github/workflows/sync.yml](.github/workflows/sync.yml). Add `GITLAB_TOKEN` as a
repository secret (and grant the workflow token repo access) and it runs every
6 hours.

Plain cron works too:

```cron
0 */6 * * * cd /opt/githublab-sync && GITHUB_TOKEN=… GITLAB_TOKEN=… githublab-sync sync >> sync.log 2>&1
```

## Using the Claude Code agent

If you use [Claude Code](https://claude.com/claude-code), the bundled subagent
at [.claude/agents/repo-sync.md](.claude/agents/repo-sync.md) is auto-discovered.
Just ask, e.g.:

> "Sync my repos between GitHub and GitLab" · "Back up GitHub to GitLab" ·
> "Set up double backup between the two providers"

The agent runs `doctor`, previews with `--dry-run`, executes the sync, and walks
you through any diverged branches — always committing as a human (see below).

## Commit attribution policy

This is a backup tool, so anything it produces reads as if a human wrote it:

- The CLI **never** adds AI attribution to anything it creates.
- With `strip_ai_attribution: true`, AI markers (`Co-Authored-By: Claude`,
  `🤖 Generated with [Claude Code]`, `Assisted-by`, etc.) are removed from
  mirrored PR/MR titles and descriptions.
- The Claude Code agent is explicitly instructed to write plain, human commit
  messages with no AI references.

Git **history is mirrored verbatim** by default (rewriting history changes commit
hashes and is destructive). An opt-in `scrub_commit_messages` helper exists for
those who explicitly want to rewrite historical commit messages.

## Conflict handling

When a branch has diverged (commits unique to *both* sides), GitHubLab Sync
**will not** pick a winner or force-push — it reports the conflict and exits `3`.
Resolve it yourself with a normal merge/rebase, then re-run the sync. This is the
safety guarantee that makes bidirectional mirroring trustworthy.

## FAQ

**Does it copy issues, wikis, or CI config?** No — it focuses on git data
(branches, tags) and open PRs/MRs, which is what you need to restore a project.

**Is bidirectional safe?** Yes. Only fast-forward updates are pushed
automatically; anything ambiguous becomes a reported conflict.

**Can I run one-way only?** Set `direction: github-to-gitlab` or
`gitlab-to-github` (destination branches are force-matched to the source).

**Does it work with self-hosted GitLab / GitHub Enterprise?** Yes — set `host`
and `api_url` per provider.
